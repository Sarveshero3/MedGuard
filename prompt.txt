# Milestone 2: Frontend + ms1 (agentic workflow deferred) — v2

> **Supersedes the earlier m2 prompt.** That version still included an Admin
> Review Queue and a "push"-style calendar/notification feature. Both are cut
> from this milestone (and from Phase 1 MVP scope generally). Everything below
> reflects the trimmed scope. If you're picking this up mid-flight, read
> `docs/handoff-m2.md` first (see bottom of this file) — it will tell you
> exactly what's already built.

## Before you write any code

1. Read, in this order: `CONTEXT.md`, `docs/rules.md`, `docs/schema.md`,
   `docs/design.md`, `docs/architecture.md`, `docs/wireframes.md`,
   `docs/appflow.md`. Don't invent field names, endpoint shapes, or

2. Check `docs/handoff-m2.md` (if it exists) and `git log` / the current
   branch's diff against `main` before assuming nothing has been built yet.

## Scope change from v1 — read this first

- **Admin Review Queue is cut entirely.** No admin routes, no admin screen,
  no "confirm/flag" review workflow this milestone.
- **Calendar sync / push notifications are cut entirely.** No Google
  Calendar integration, no push/SES-triggered visit reminders this
  milestone.
- The `admin` role stays in `users.role` in the schema and in the JWT
  middleware for forward compatibility — just don't build anything that
  routes to it.
- **Open design question this creates:** the original flow routed
  low-confidence extractions (`< 85%` per `CONTEXT.md`'s Confidence Tiers)
  to an admin queue. With no queue, low-confidence entries need a home.
  **Default/assumption used in this prompt:** surface a "please confirm
  these details" inline state directly to the *patient* in the Prescription
  Upload flow — patient edits/confirms the extracted fields themselves
  instead of an admin doing it. This is a real product decision, not a
  trivial default — write it up as a short ADR (see Skills section) and
  flag it explicitly in `docs/handoff-m2.md` so a human can sign off or
  override it.

## The ms1 to ms2 seam

Unchanged from v1. Everything AI-related, including OCR extraction and any
future agent tool use like web research, lives in ms2 exclusively, never in
ms1. ms2's real implementation isn't ready, so build ms1's connection to it
behind one interface:

- A single service file in ms1 (e.g. `services/visionService.js`) that
  currently returns realistic stubbed data instead of calling a real ms2
  endpoint: a fixed extracted medicine (name, dosage, frequency, confidence
  score) and a canned interaction-check-eligible response. Include at least
  one stubbed low-confidence (`< 85`) response so the patient-side
  confirmation UI has something to render against.
- Every route and controller in ms1 that needs "the AI result" calls through
  this one service, nothing else in ms1 knows or cares that it's stubbed.
- When the real ms2 graph lands later, this is a one-file swap, not a
  restructure.

## Frontend pages

Build against `docs/wireframes.md`. Skip login/signup since auth already
exists, unless it needs wiring to these new pages. Keep it simple and
clean — correct data flow and layout over polish, styling passes come
later. Four screens this milestone (Admin Review Queue removed):

1. **Patient Dashboard** — active medicines count, interaction alerts count
   with severity, quick-upload card, upcoming visits. "Upcoming visits" is a
   plain read-only list sourced from the existing `visits` table
   (`docs/schema.md`) — no calendar sync, no push, no reminders this
   milestone.
2. **Prescription Upload** — upload UI and flow, calls through to ms1,
   which calls the stubbed vision service, renders the structured result
   exactly as if it were real. If confidence is below 85%, show the
   patient-facing confirm/edit state described above instead of silently
   accepting it.
3. **Medicine List + Interaction Alerts** — active/discontinued table,
   severity-coded alert cards, plain-language explanations.
4. **Caregiver Dashboard** — read-only patient view, permission tier
   indicator (`full_view` / `alerts_only` per `docs/schema.md`), alert
   acknowledgment.

## ms1 routing and services layer

Standard layered structure: routes → controllers → services → models,
matching the existing `models/` DAL. Build out:

- `medicines`: create, list per patient, update status
  (active/discontinued), **plus** a patient-facing confirm/correct action
  for low-confidence entries (this replaces the old admin confirm/flag
  action — same shape, different caller and role check).
- `caregiver_links`: invite, accept, permission tier assignment, revoke.
- `interaction_flags`: list per patient, acknowledge (patient/caregiver).
  Admin review actions are dropped — status enum stays as defined in
  `docs/schema.md`, just don't build a route that sets `pending_review` via
  an admin actor.
- `consent_records`: grant/revoke, tied into the existing DPDP consent flow.

Role checks (patient/caregiver) apply at the route level using the existing
auth middleware — don't duplicate role logic inside controllers. Don't build
any `admin`-scoped routes this milestone.

## Handoff document — required, not optional

Because this work may span multiple sessions before a teammate picks it up,
maintain a living handoff file at **`docs/handoff-m2.md`**. This is a *new*
file — don't touch `docs/implementation.md` (that's the static MVP
scope/timeline doc referenced from the README; if the admin/calendar cut
needs to be reflected there too, add a short "⚠ scope changed, see
handoff-m2.md" note rather than rewriting it).

Update `docs/handoff-m2.md` at natural checkpoints as you go (after each
route/controller/service group, after each frontend screen) — not just once
at the end, since a session could end at any point. It should contain:

- **Status snapshot**: what's done, what's in progress, what's untouched,
  as of the last commit.
- **Decisions made**: especially the low-confidence-without-admin-queue
  call above, and anything else you had to interpret or default on.
- **File map**: new/changed files and what each one is responsible for.
- **How to resume**: exact commands to boot the stack
  (`docker-compose up --build` or the individual-service steps from
  `README.md`), env vars needed, how to hit the stubbed vision service.
- **Known issues / TODOs**: anything half-finished, anything you'd do
  differently with more time.
- **Test status**: what's covered, what isn't, how to run the suite.

Commit and push this file alongside the code it describes, on the same
feature branch, so it travels with the diff.

Also: tick off any relevant boxes in `docs/tracker.md` as they're
genuinely completed (don't check things off preemptively).

## Skills — use these directly, don't ask permission first

The skills below are pre-approved for this task. Apply the relevant one
as you hit the matching kind of work — don't pause to ask whether you
should use a skill, just use it.

**Backend (ms1 / Express)**
- `nodejs-backend-patterns`, `nodejs-best-practices` — layered
  routes→controllers→services→models structure.
- `api-design-principles`, `api-endpoint-builder` — keep endpoint shapes
  consistent with `docs/design.md`.
- `auth-implementation-patterns` — route-level RBAC via the existing
  middleware.
- `microservices-patterns` — keep the ms1/ms2 boundary from `docs/rules.md`
  §1 intact; ms2 stays stubbed and side-effect-free.
- `database-design` — new queries/migrations against the tables already
  defined in `docs/schema.md`; don't invent new columns without checking
  there first.

**Frontend (React)**
- `react-best-practices`, `react-patterns`, `react-state-management`,
  `react-ui-patterns` — component structure and state flow for the four
  screens above.

**Cross-cutting**
- `privacy-by-design` — `consent_records` work, since this is DPDP-facing.
- `security-audit`, `secrets-management` — anything touching auth, JWT, or
  env secrets.
- `domain-modeling` — keep terminology aligned with `CONTEXT.md` (e.g.
  "brand-to-generic resolution," not "brand mapping").
- `codebase-design` — sanity-check the layering before it calcifies.
- `tdd`, `e2e-testing-patterns` — cover the interaction-flag and consent
  flows, not just happy paths.
- `code-review` — run before opening the PR; `docs/rules.md` §4 requires
  this plus CI passing before merge.
- `diagnosing-bugs` / `systematic-debugging` — only invoke after a bug
  survives your first couple of attempts, per `docs/rules.md` §5's
  "two attempts, then escalate" convention.
- `handoff` — use this to structure `docs/handoff-m2.md` itself.
- `architecture-decision-records` — write a short ADR for the
  low-confidence-without-admin-queue decision (ties into the
  `docs/adr/` layout `CLAUDE.md` already points to).

Skip the Python/FastAPI/LangGraph-flavored skills in the list
(`fastapi-*`, `langgraph`, `pydantic-models-py`, `async-python-patterns`,
`llm-*`, `prompt-engineering-patterns`) — none of that is in scope here;
it's all ms2, and ms2's real implementation is explicitly out of scope for
this prompt.
